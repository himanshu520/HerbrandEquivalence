◆ findAvailableVariables()

void HerbrandPass::findAvailableVariables ( Function &  F)

Finds available variables at each instruction of a function.

Calculation of availVariables is a forward dataflow problem, calculated via fixed point computation. A variable is available at a point if all paths from the start of the program to that point contains a definition of the variable. Our universe is U - the set of program variables and for each program point P , we initialise OUT[P] to U . Also, let GEN[P] to be the variable defined at P if any, otherwise empty set. We will perform updates as

IN[P] = intersect(availVariables[P']) OUT[P] = union(IN[P], GEN[P])

where union is normal set union and intersect is performed over all P' belonging to the predecessor of P . Finally, availVariables[P] = OUT[P] .
For the purpose of implementation, IN and OUT will not be maintained explicitly, but the same information will be kept in availVariables itself.

Parameters
[in]FFunction under consideration
Returns
Void
Note
Available Variables is different from both available expressions and reaching definition.

◆ constIDStruct

map<int, IDstruct *> HerbrandPass::constIDStruct

Stores the pointer to IDstruct object corresponding to a constant, so that we never create a duplicate IDstruct object for the same.

This is required just for logical consistency. A basic block other than the starting block can have no predecessor (in case a block of code is unreachable). In that case, to make the code also useful for GVN along with herbrand equivalence, this information is required. Otherwise at the start of all such unreachable basic blocks we would be creating a new IDstruct object for the same constant - which means according to our program, they are not equivalent (as they do not point to the same IDstruct object), which is not correct. However, for herbrand equivalence we can even do without this.

◆ Parent

map<tuple<char, IDstruct *, IDstruct *>, IDstruct *> HerbrandPass::Parent

For a given {op, leftID, rightID} as key, stores pointer to IDstruct object whose fields are {op, _, leftID, rightID}.

When we create an IDstruct object with fields {op, _, leftID, rightID}, we store this information in the map so that when we again need such an object we do not create it again and use the already existing one.

◆ partitions

map<Instruction *, Partition> HerbrandPass::partitions

Stores Herbrand Equivalence information after each instruction in the program.

Each index of Partition, corresponds to either a constant or a variable or a two operand expression as determined by indexCV and indexExp. There will be an IDstruct object for each program point and those variables/constants/expressions that point to the same IDstruct object (has same pointer value) at a given program point are Herbrand Equivalent at that program point.

See also
indexCV, indexExp, Partition
